============================================================================
                           MIXIN PATTERN (STRUCTURAL)
============================================================================

Intent:
The Mixin pattern is a design strategy used to add specific behaviors to 
a class hierarchy through "Horizontal Composition". Unlike traditional 
inheritance, Mixins allow a class to pick and choose multiple independent 
features at once.

The Problem:
Standard inheritance often leads to rigid and deep trees. If you want a 
"Dragon" that can 'Fly' and 'Shoot Laser', and an "Elephant" that can 
'Walk' and 'Shoot Gun', a single inheritance tree would require duplicate 
code or a very complex, brittle structure.

The Solution:
In C++, Mixin Inheritance is achieved by inheriting from template 
arguments. Specifically, using Variadic Templates (C++11/17), we can 
create a generic 'Entity' class that inherits from a pack of 'Mixins' 
simultaneously.

Mixin vs. Decorator:
- Decorator (Vertical Composition): Wraps an object inside another
  (Layered/Recursive). Ideal for modifying existing behavior at runtime
  (see 11_Decorator/Static_Mixin).
- Mixin (Horizontal Composition): Inherits from multiple classes at the 
  same level (Parallel). Ideal for composing a final type with a fixed set 
  of features at compile-time (see 26_Strategy/03_Static_Strategy).

Our Examples:
This folder contains two levels of complexity:

1. Simple/:
   Demonstrates basic horizontal composition. We define features like 'Fly', 
   'Walk', 'Laser', and 'Gun'. Then, we use template aliases (using) to 
   instantiate specific entities like 'Dragon' or 'Airplane' by simply
   listing their required capabilities.

2. Advanced/:
   Introduces inter-mixin communication and advanced C++ features:
   - Dependency: The 'Fly' mixin checks the 'Tank' mixin for fuel.
   - Static Visitor Pattern: Implements a 'visitFeatures' method using 
     Fold Expressions (C++17) to iterate over all inherited bases at 
     compile-time, replacing virtual tables with static dispatch
     (see 28_Visitor).
   - Efficiency: Demonstrates how to handle a pack of base classes 
     statically without losing the ability to process them as a group.

Key Benefits:
- Granular Reusability: Small, focused classes (Mixins) can be combined 
  in infinite ways.
- Zero Overhead: The compiler flattens the hierarchy, allowing for 
  aggressive inlining and eliminating virtual function overhead.
- Code Clarity: The resulting types (e.g., Entity<Fly, Tank, Laser>) 
  document exactly what the object can do.
